# Chest Cavity Legacy 架构重构计划

## 总目标

把项目从“固定 27 格物品栏 + 字符串 score Map + Forge 事件散落逻辑 + CrT 直接改旧 registry”重构为：

- 内容层：JSON / builtin / CrT 都只声明 `OrganDef`、`BodyTypeDef`、`ChestLayoutDef`、ability spec、effect spec。
- 编译层：reload 后统一生成 `CompiledContent`，运行期只读。
- 状态层：`ChestCavityData` 只保存实体胸腔状态、slot stack、opened、timer、runtime version。
- 运行时层：`ChestCavityRuntime` 是由状态 + compiled content 构建出的 immutable snapshot。
- 能力向量：score / baseline / delta 以数组为真源，字符串 Map 只作为外部视图。
- 修改入口：实体状态写入统一通过 `ChestCavityMutations`。
- 行为入口：Forge 事件只构造 context，实际逻辑由 spec/effect pipeline 执行。
- UI：ModularUI 完全由 layout / slotRules / snapshot 驱动。
- CrT：只做内容注册和实体 mutation，不直接绕过 compiled runtime/cache。

不再把“兼容旧内部写法”作为优先目标。旧 API 如果只是对外脚本入口，可以保留薄壳；如果只是内部历史包袱，直接删除或改写。

## 当前真实状态校准

### 1. 状态与 runtime 底座：核心完成

已完成：

- Forge capability 已直接承载 `ChestCavityData`。
- `ChestCavityHelper` 已收缩为薄门面。
- closed entity 不再常驻 27 个空 `ItemStack`。
- 未开胸实体序列化不写空 `Inventory`。
- `OrganInstance` 已加入 slot 级缓存。
- `ChestCavityRuntime.rebuild` 已消费 `OrganInstance`，不再在 runtime 内反复解析 `ItemStack -> OrganData`。
- `ChestCavityRuntime` 已有 score / baseline / delta。
- score / baseline / delta 已以 `float[]` 为运行时真源。
- `Map<String, Float>` 已变成 API / UI / CrT 需要时懒生成的视图。

本轮清理：

- 已删除 `ChestCavityRuntime.withScoreValues(Map<String, Float>)` 兼容式 runtime score 覆盖入口。
- 已删除 `ChestCavityMutations#setScore/addScore` 和 `ChestCavityView#setOrganScore/addOrganScore` 这条临时覆盖链路。
- 已删除旧 `OrganManager` / `OrganSerializer` 外壳，JSON organ 解析并入 `DataLoaders`，客户端 organ 内容同步改由 `ContentSync` 承担。
- 已将 `ChestCavityUiBridge` 收束为普通 UI 服务类 `ChestCavityUi`。

仍需清理：

- `ChestCavityData#setRuntimeCommitted` 为比较 score version 仍会生成 Map 视图；后续应改为数组版本比较或 dirty bit。
- 仍有大量事件通过字符串 score id 查询，需要继续切到 `ScoreRef` 或 compiled spec。

结论：

- 状态/runtime 的第一阶段已经成立。
- runtime 的内容输入已经切到 `CompiledContent`。
- 下一步不是继续补兼容壳，而是把事件行为、slotRules、UI 同步接入 compiled content。

### 2. mutation 入口统一：内部主链路完成，外部兼容壳仍存在

已完成：

- 新增 `ChestCavityMutations`。
- `ChestCavityData` 删除旧 public 写入口：
  - `setOrgan`
  - `setOrganAndRecalculate`
  - `setOrganScore`
  - `addOrganScore`
  - `setOrganScoreAndSync`
  - `addOrganScoreAndSync`
  - `recalculateScores`
  - `recalculateAndSync`
  - `applyAndSyncScoreChanges`
- UI / CrT view / ItemHandler / 掉落 / 玩家死亡重置已改为通过 mutation 提交变更。
- runtime rebuild、score side effects、version 递增、owner sync 已集中到 `Mutation#commit`。
- 批量修改可以在一次 transaction 中完成，避免多次 rebuild/sync。

本轮清理：

- 已删除 `ChestCavityView#setOrganScore/addOrganScore`。
- 已删除 `ChestCavityMutations#setScore/addScore`。
- `ChestCavityView#recalculateScores` 仍作为 CrT 可见的薄刷新入口保留，内部走 mutation。

仍需清理：

- score side effects 仍硬编码在 mutation commit 中，后续应改为 mutation result event 或 effect pipeline。

结论：

- 实体 slot 写入主链路已经统一。
- score 覆盖能力还不是最终形态，后续要决定：删除、限制为 debug、或改为 transient modifier。

### 3. score / ability / effect：大块 B 已完成

已完成：

- 新增 `ScoreRef`。
- `CCOrganScores` 保留字符串常量，并新增 `*_REF` typed ref。
- `ScoreApi` 支持 indexed score 注册与查询。
- runtime 支持 `getScoreValue(ScoreRef)` / `getBaselineScoreValue(ScoreRef)` / `getDeltaScoreValue(ScoreRef)`。
- attribute 修改已改为 `AttributeEffectSpec[]`。
- 主动投射物能力已 spec 化：
  - pyromancy
  - dragon_bombs
  - forceful_spit
  - ghastly
  - shulker_bullets
- 新增 `BodyEventContext` / `TickEffect`。
- passive tick 已迁移到 `PASSIVE_TICK_EFFECTS`：
  - glowing
  - hydroallergenic
  - hydrophobia
  - lightweight
  - buoyancy
  - photosynthesis
  - crystalsynthesis
  - organ rejection
- `CombatScoreEvents` 已改成 spec pipeline：
  - attack cancel specs
  - damage modifier specs
  - hit effect specs
  - damage side-effect specs
- `FoodScoreEvents` 已改成统一 nutrition profile。
- 食物分类逻辑已取消：
  - meat / plant / rotten / furnace 分类不再参与营养计算
  - carnivorous / herbivorous / rotgut / rot digestion score 已从默认 registry 和内置数据中移除
  - 胃类器官统一提供 `digestion`
  - 肠类器官统一提供 `nutrition`
- `PotionScoreEvents` 已改成 potion adjustment spec pipeline。
- venom effect 查询已改为 runtime slot index，不再遍历 stack 反查 organ data。

仍未完成：

- effect/spec 目前是 builtin Java spec 数组，还没有开放成 JSON/CrT 数据声明。

结论：

- 事件层已经完成 pipeline 化。
- 后续若继续扩展，应把 builtin spec 注册到 `ContentManifest`，而不是再写新的 Forge event 分支。

### 4. 内容模型与 CrT：大块迁移 A 已完成

- 新增 `ContentRegistry`。
- 新增 `OrganDef` / `BodyTypeDef` / `ExceptionalOrganDef`。
- 新增 `CompiledContent`。

- CrT 新增 `mods.chestcavity.ChestLayout.registerGrid(...)`。

- JSON organs / types / layouts / entity_assignment 已统一写入 reload manifest。
- CrT organ/type/layout/entity assignment 已统一写入 manifest operation。
- `DataLoaders.reload` 已改为：
  - 创建空 reload manifest
  - 加载 classpath JSON
  - 加载 config JSON
  - 重放 CrT/script manifest operation
  - compile 成 `CompiledContent`
  - publish runtime snapshot
- 删除旧 `RUNTIME_OVERRIDES` 重放机制。
- 删除旧 `GeneratedChestCavityType`。
- 删除旧 `FallbackChestCavityType`。
- 删除 `OrganData` 静态 registry。
- `OrganData.get(ResourceLocation/ItemStack)` 已改为读 `CompiledContent`，旧 `fromRegistry` 命名已删除。
- `DataLoaders.getType/getTypes/getEntityAssignments` 已改为读 `CompiledContent`。
- `ChestCavityTypeUtil#getChestLayout` 已改为读 compiled layout。

真实问题：

- compiled content 当前仍通过 `ChestCavityType` 接口承接 runtime，没有把 runtime 直接改成 `BodyTypeDef` / compiled body type。
- ability / effect spec 还未进入 manifest。
- drop table 仍在 `DropApi` 自己的内存表，不属于 compiled content。

结论：

- 内容层已经从“多个旧 registry”收束到 “manifest operation -> CompiledContent”。
- 下一步应把 event spec / slotRules / UI snapshot 接入这个 compiled content。

### 5. layout / ModularUI：slotRules 已完成

已完成：

- 新增 `ChestLayoutDef`。

- ModularUI 面板尺寸、标题位置、slot 数量、slot 坐标由 layout 驱动。

  - `slotSpacingX` / `slotSpacingY`
  - `migrationStrategy`
  - `forbiddenSlots`
- layout 已升级为 per-slot `SlotRule`。
- slotRules 已支持：
  - forbidden
  - allowedItems
  - allowedScores
  - minStackSize
  - maxStackSize
- JSON `slotRules` 可声明 slot rule。
- CrT `ChestLayout.setSlotRule(...)` 可声明 slot rule。

结论：

### 6. 网络 / UI snapshot：基础完成

当前状态：

- 新增 `BodyUiSnapshot`。
- `ChestCavityNetwork.sendChestCavitySync` 已发送 snapshot NBT。
- snapshot 包含：
  - state
  - layout id
  - slot count
  - slot stacks
  - slot rules summary
  - score summary
  - organ/runtime/score version
- client sync 从 snapshot 中取 state 应用到 capability。
- `organVersion` / `runtimeVersion` / `scoreVersion` 已进入 snapshot。

仍未完成：

- delta sync 可在 snapshot 基础上继续拆细，不需要再改状态模型。

## 剩余核心工作

### 大块迁移 A：内容编译层 + CrT 注册路径重写：已完成

已完成目标：

- 让 JSON / builtin / CrT 都只写 `ContentManifest`。
- 新增 `CompiledContent` 作为运行期只读内容源。
- `DataLoaders.reload` 从“边读边改 registry”变成“读取 manifest -> compile -> publish snapshot”。
- 删除 `RUNTIME_OVERRIDES` 重放机制。
- 让 CrT 内容注册不再直接写 `OrganData` / `GeneratedChestCavityType` / `ChestLayouts`。

已落地逻辑：

1. 扩展 `ContentManifest`
   - 增加 organs。
   - 增加 body types。
   - 增加 entity assignments。
   - 增加 layouts。

2. 新增 definition 类
   - `OrganDef`
   - `BodyTypeDef`
   - `ExceptionalOrganDef`
   - layout 继续使用 `ChestLayoutDef`。

3. 新增 `CompiledContent`
   - 持有 immutable maps：
     - item id -> compiled organ
     - type id -> compiled body type
     - entity id -> type id
     - layout id -> layout
   - 提供 fallback type。
   - 提供按 item / ore / type 查询的只读方法。

4. 改造 `DataLoaders`
   - reload 开始创建新的 `ContentManifest`。
   - JSON 读取只写 manifest。
   - CrT 注册源写 manifest operation。
   - reload 结束 compile manifest。
   - publish compiled snapshot。

5. 旧接口过渡方式
   - `DataLoaders.getType/getTypes/getEntityAssignments` 改为读 `CompiledContent`。
   - 新增 compiled-backed `ChestCavityType` implementation。
   - `OrganData.fromStack/get` 的运行期查询改为读 compiled organ。
   - 删除旧 register/unregister 静态 registry。

6. CrT 改造
   - `OrganDataApi` 写 manifest organ def。
   - `ChestCavityTypeApi` 写 manifest body type operation。
   - `LayoutApi` 写 manifest layout def。
   - `EntityAssignmentApi` 写 manifest assignment operation。
   - 删除 `DataLoaders.applyRuntimeOverride`。

相较旧版优点：

- reload 顺序清晰：声明 -> 编译 -> 发布。
- CrT 不再需要 runtime override replay。
- 数据源统一，JSON 和 CrT 不会分别改不同 registry。
- runtime 查询只读 snapshot，更容易缓存、更容易同步、更少并发/重载风险。
- `GeneratedChestCavityType` 已被 compiled-backed 类型替换，减少一大块状态型类的心智负担。

验收：

- `.\gradlew.bat compileJava` 通过。
- JSON organs/types/layouts/entity_assignment 仍能加载。
- CrT 注册 organ/type/layout 后 reload 生效。
- `ChestCavityTypeUtil` 查询结果不变。
- `ChestCavityRuntime.rebuild` 从 compiled-backed content 获得相同 score。

### 大块迁移 B：已完成

- Forge 事件层只负责拿 context 和分发。
- Damage/Food/Potion 行为收束成 spec。
- layout 从“位置定义”升级为“胸腔规则定义”。
- UI 从 capability NBT 高频同步改为 snapshot/delta。
- slotRules 已进入 layout / manifest / compiled content 路径，并开放给 CrT。

当前边界：

- 大块 A 已经提供稳定内容编译层。
- B 不需要再处理 organ/type/layout 的旧 registry 问题。
- slotRules 已完成。
- Damage/Food/Potion 已 pipeline 化。
- UI snapshot 基础已完成。
- ability/effect spec 的 JSON/CrT 数据化注册仍是后续扩展项，不属于本轮必要架构阻塞。

具体逻辑：

1. slotRules
   - `ChestLayoutDef` 增加 per-slot rule：
     - forbidden
     - allowed item ids
     - allowed score refs
     - min/max stack size 可选
   - JSON / CrT 可注册 slot rules。
   - `CompiledContent` 编译 layout slotRules。已完成。
   - `ChestCavityTypeUtil` 提供统一 `canPlaceOrgan(chestCavity, slot, stack)`。
   - `OrganItemHandler` 和 ModularUI 都调用同一个规则函数。

2. Damage spec
   - 把 `CombatScoreEvents` 拆成：
     - incoming damage modifiers
     - attack hit effects
     - damage finalization effects
     - collision side effects
   - 每个 spec 声明：
     - score ref
     - baseline/delta/absolute 读取方式
     - trigger 条件
     - formula
     - side effect

3. Food spec

4. Potion spec
   - 把 beneficial / bad / wither / poison 条件抽成 matcher。
   - duration factor 变成 potion adjustment spec。
   - 反射修改 duration 只留在 pipeline 末端。

5. UI snapshot / delta sync
   - 新增 `BodyUiSnapshot`。
   - snapshot 包含：
     - entity id
     - layout id / layout version
     - slot stacks
     - forbidden/allowed state summary
     - score summary
     - organ/runtime/score version
   - snapshot 基础已完成。

相较旧版优点：

- 事件行为能被 CrT/JSON 数据化扩展。

- `OrganItemHandler`、UI、mutation 使用同一套 slot rule，减少规则不一致。
